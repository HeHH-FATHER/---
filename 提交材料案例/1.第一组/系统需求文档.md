# 原神数据决策系统 需求文档

> 版本：v2.0 | 日期：2026-07-12 | 小组：第一组

---

## 1. 项目概述

### 1.1 系统目标

随着《原神》游戏持续运营，深境螺旋（深渊）玩法已成为衡量角色强度与版本Meta变化的核心场景。国内玩家社区"提瓦特小助手"每期收集数十万份深渊挑战数据，覆盖60余个版本、超过110个角色。然而，这些数据均为聚合统计，缺乏个体粒度的分析与流式实时监控能力。

本系统基于上述真实统计数据，构建一套完整的大数据采集、离线批量分析、实时流计算、AI智能分析与可视化展示平台。核心思路采用"统计→模拟→聚合→验证"闭环：利用聚合统计数据反推个体玩家记录，经大数据管道处理后，回归对比原始统计，以验证数据管道的正确性。

### 1.2 职责与范围

#### 1.2.1 功能范围

本系统覆盖以下核心功能模块：

1. **数据生成**：基于提瓦特小助手API聚合统计数据，反推生成个体玩家深渊记录（含角色BOX与深渊战绩）、抽卡记录（含保底机制）、练度记录（含武器/圣遗物/命座）和满意度投票记录。支持脏数据注入以验证清洗规则有效性。
2. **离线数据管道**：采用Hadoop MapReduce构建四层数据仓库（ODS→DWD→DWS→ADS），实现深渊管道（10条清洗规则）和抽卡管道（五星过滤+物品聚合），多维度聚合与MySQL结果层加载。
3. **实时流计算**：基于Kafka + Spark Streaming构建两条实时链路——练度数据（build-v2）与满意度数据（satisfaction-v1）——通过滑动窗口聚合至Redis，支撑大屏实时面板。
4. **数据大屏**：暗色科技风三列网格布局，7个翻牌器KPI + 8个面板（满意度排行、热门练度、持有使用率TOP15、卡池占比、热门配队TOP8、版本涨跌、链路监控、练度滚动日志），6节点系统资源实时监控。
5. **数据分析**：10+页面纯离线数据深度钻取，包含角色排名总览、配队分析、卡池分析、角色练度深度分析、持有-使用流失分析、数据质量报告（抽卡+深渊）、历代趋势对比。
6. **AI智能分析**：基于大语言模型（DeepSeek R1-8B/硅基流动API）的智能问答模块，用户自然语言提问，系统自动查询MySQL相关数据表组装上下文，返回数据驱动的分析建议（角色推荐/配队策略/投入优先级）。
7. **数据管理**：基于RuoYi代码生成器构建6张核心数据表的完整CRUD管理界面，包含3张DIM维表（角色/武器/圣遗物）与5+张ADS结果表。

#### 1.2.3 用户群体

- **游戏运营人员**：查看角色使用率排名、配队趋势变化、版本Meta演变，辅助游戏运营决策。
- **数据工程师**：监控大数据管道的清洗通过率、脏数据分布，验证离线与实时数据的一致性。
- **玩家社区**：了解版本强势角色、热门配队组合、卡池抽取热度分布。使用AI智能分析获取个性化配队建议。

#### 1.2.4 系统边界

**包含**：
- 离线数据四层管道（深渊+抽卡两条链路）
- 实时数据两条链路（练度/满意度 → Kafka → Spark Streaming → Redis → 大屏）
- 数据大屏（实时为主，离线为辅，7KPI+8面板）
- 数据分析页面（10+页面，纯离线数据）
- AI智能分析（自然语言问答，数据库驱动上下文）
- 数据管理CRUD

**不包含**：
- 真实游戏数据采集SDK（本项目为模拟数据）
- 多租户数据隔离
- 移动端适配

## 2. 功能需求

### 2.1 数据生成模块

| 编号 | 功能 | 描述 |
|------|------|------|
| GEN-01 | 深渊记录生成 | 基于v6.6深渊统计反推200人个体记录（含BOX+战绩），5%脏数据注入 |
| GEN-02 | 抽卡记录生成 | 含保底机制(74/90/50-50)，版本时间区间内日期随机化，Kafka+文件双输出 |
| GEN-03 | 练度记录生成 | 122角色基底，sqrt权重分配，50条/秒持续产出 |
| GEN-04 | 满意度生成 | 121角色，3秒一轮，综合/能力/外观三维评分 |

### 2.2 离线数据管道

| 编号 | 功能 | 描述 |
|------|------|------|
| PIPE-01 | 深渊ODS | 扫描生成器JSON文件，配对合并为JSONL→HDFS |
| PIPE-02 | 深渊DWD | 10条清洗规则（星级/命座/等级/字段缺失/BOX空/队伍重叠/角色约束/人数/UID/Teams） |
| PIPE-03 | 深渊DWS | 角色指标聚合 + 配队共现统计 |
| PIPE-04 | 深渊ADS | MySQL写入ads_char_summary/ads_team_usage/ads_team_matrix |
| PIPE-05 | 抽卡ODS | preprocess_gacha.py处理逐抽记录→HDFS |
| PIPE-06 | 抽卡DWD | CleanGachaMR过滤五星，丢弃3★/4★ |
| PIPE-07 | 抽卡DWS | GachaAggMR按物品聚合COUNT |
| PIPE-08 | 抽卡ADS | load_gacha_to_mysql.py写入ads_gacha_offline |

### 2.3 实时流计算

| 编号 | 功能 | 描述 |
|------|------|------|
| RT-01 | 练度消费 | Spark Streaming 1s batch，60s窗口TOP4→Redis build:hot_chars |
| RT-02 | 满意度消费 | Spark Streaming 3s batch，ZADD→Redis ZSET排行，LIST趋势，STRING快照 |
| RT-03 | Kafka管理 | 2个Topic（build-v2/satisfaction-v1），consumer group每次重启唯一化 |

### 2.4 数据大屏

| 编号 | 功能 | 描述 |
|------|------|------|
| SCR-01 | KPI翻牌器 | 7个核心指标（抽卡量/出金/五星率/角色金/窗口角色数/满意度波动/数据新鲜度） |
| SCR-02 | 满意度面板 | TOP6实时排名（综合/能力/外观+涨跌+趋势迷你图），5s轮询 |
| SCR-03 | 热门练度面板 | TOP4角色头像+武器使用率+圣遗物图标+均伤，3s轮询 |
| SCR-04 | 持有使用率TOP15 | 头像+双柱图（持有率底+使用率顶），离线数据 |
| SCR-05 | 卡池占比 | 角色/武器双环形图，5min轮询 |
| SCR-06 | 热门配队TOP8 | 队伍名+角色头像+使用率/持有率 |
| SCR-07 | 版本涨跌 | 哑铃图（prev rate vs curr rate），5min轮询 |
| SCR-08 | 链路监控 | 生成器/Spark状态 + Kafka消息量 + Redis存储 + 6节点CPU/MEM |

## 3. 非功能需求

| 编号 | 类别 | 要求 |
|------|------|------|
| NF-01 | 性能 | 离线管道单批次≤10分钟（深渊200人/抽卡5000抽）；大屏面板首次加载≤3s |
| NF-02 | 可用性 | 6节点一键启停（start_all.sh/stop_all.sh），异常自动恢复 |
| NF-03 | 可维护性 | 模块化Shell脚本，日志分离（/tmp/*.log），代码Git版本管理 |
| NF-04 | 兼容性 | 支持Windows 11（开发）+ CentOS 7（部署），JDK 1.8 |
| NF-05 | 安全性 | RuoYi框架权限管理，@Anonymous注解允许大屏/AI匿名访问 |
| NF-06 | 数据一致性 | 离线管道Redis时间戳同步，Spark Streaming offset管理 |
