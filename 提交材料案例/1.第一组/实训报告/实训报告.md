## 一、实训概述

### 1.1 项目名称

**原神数据决策系统** — 基于Hadoop/Spark生态的游戏大数据分析平台

### 1.2 项目背景与意义

本次实训以《原神》游戏深境螺旋（深渊）玩法为业务场景，构建一套覆盖数据生成、离线批量分析、实时流计算、AI智能分析与可视化展示的完整大数据系统。项目采用"统计→模拟→聚合→验证"闭环思路，从真实社区API聚合统计数据出发，反推个体玩家记录，经大数据管道处理后回归对比原始统计，验证数据处理全链路的正确性。

通过本项目的设计与实现，综合运用了《Hadoop编程实践》、《MapReduce编程实践》、《Spark编程》、《Spark Streaming流式计算》、《中间件实践》、《RuoYi框架实践》、《实时大屏实践》、《AI大模型实践》等课程所授知识与技能，完成了从底层集群部署到上层AI分析大屏的端到端大数据系统开发。

### 2.2 第二周：离线数据管道开发（6.30 — 7.03）

**主要任务（何宏昊 - 深渊链路）**：

1. **数据生成器开发（Abyss-Record-Generator）**：基于Jackson JSON解析与伯努利采样算法，从聚合统计数据反推个体玩家深渊记录。核心实现包括：按own_rate做伯努利试验决定角色拥有关系、预计算合法配队组合、加权随机配队选择、命座分配策略、10种脏数据注入机制。支持命令行参数配置（--users、--seed、--dirty-rate）。

2. **ODS层开发（preprocess_abyss.py）**：扫描生成器产出的每用户两个JSON文件，配对合并为单行JSONL格式，上传至HDFS /data/abyss/ods/。

3. **DWD层开发（AbyssCleanMR.java）**：Hadoop MapReduce Map-only作业，使用Jackson解析JSON，逐条执行10条清洗规则校验（星级/命座/等级/关键字段/BOX非空/队伍重叠/角色约束/队伍人数/UID/Teams结构）。通过MultipleOutputs实现多路输出——干净数据写入dwd_char_detail与dwd_team_usage，脏数据按类型分目录归档。

4. **DWS层开发（AbyssAggMR.java）**：MapReduce作业，自定义VersionPartitioner确保同版本数据进入同一Reducer。同步开发compute_team_cooccur.py计算队内角色两两共现次数。

5. **ADS层开发（load_ads_to_mysql.py）**：通过PyMySQL写入MySQL三张ADS大屏消费表。

6. **全链路联调**：200条输入 → 187条通过清洗（93.5%）→ 14,635行DWD角色明细 → 115角色DWS聚合 → 65条ADS记录写入MySQL。

**技能收获**：MapReduce编程模型（Mapper/Reducer/Partitioner）、MultipleOutputs多路输出、Jackson JSON解析、数据仓库四层分层设计思想。

**主要任务（侯逸丞 - 抽卡链路）**：

1. **抽卡生成器开发（Gacha-Record-Generator）**：基于卡池统计JSON反推个体抽卡记录。核心实现：软保底74抽/硬保底90抽/50-50机制的自定义保底逻辑、角色命座分布（C0 63%/C2 20%/C6 6%）、武器精炼分布。支持Kafka直发+本地JSON文件双输出，自带统计验证函数。

2. **离线抽卡管道（ODS→DWD→DWS→ADS）**：
   - ODS: preprocess_gacha.py 处理生成器产出的逐抽JSONL，上传HDFS
   - DWD: CleanGachaMR.java 过滤五星出货记录，丢弃3★/4★
   - DWS: GachaAggMR.java 按物品聚合COUNT，计算五星率
   - ADS: load_gacha_to_mysql.py 写MySQL ads_gacha_offline 表

3. **卡池维表设计（dim_banner）**：录入203期卡池历史数据，关联物品名与UP版本。

**技能收获**：保底机制的概率算法设计、MapReduce聚合作业、卡池数据分析方法。

**主要任务（郭忠平 - 满意度链路）**：

1. **满意度生成器开发（SatisfactionProducerApp）**：Java独立应用，从角色满意度排行JSON读取121角色基础数据，每3秒循环生成一轮满意度投票结果，写入Kafka satisfaction-v1 Topic。支持--loop --interval参数配置。

2. **满意度消费端开发（SatisfactionStreamingConsumer）**：Spark Streaming作业（3秒batch），消费Kafka satisfaction-v1消息，逐条计算ZADD更新Redis ZSET（rt:satisfaction:ranking，121角色满意度排名），LIST维护趋势数据（rt:satisfaction:trend:*，LTRIM保留60条历史），STRING保存TOP6快照（satisfaction:top）。

3. **数据源管理**：创建satify_data.json软链接指向提瓦特数据/角色满意度排行.json，避免中文路径截断问题。

**技能收获**：Java Kafka Producer开发、Spark Streaming消费端开发、Redis ZSET/LIST/STRING数据结构实战、满意度分析业务建模。

- Spark重启读旧offset → consumer group复用 → Java每次用System.currentTimeMillis()生成唯一group
- CDN图片403防盗链 → gtimg CDN → `<meta name="referrer" content="no-referrer">`解决
- AI响应超20秒 → 前端axios timeout 20s→60s，max_tokens 4096→1024加速

- DeepSeek V4 Flash高峰期108s → 切回R1蒸馏8B平衡速度与质量

## 三、项目架构

### 3.1 系统总览

```
┌──────────────────────────────────────────────────────────────┐
│                      原神数据决策系统                          │
├──────────┬──────────┬──────────┬──────────┬──────────────────┤
│ 离线管道  │ 实时链路  │ 数据大屏  │ AI分析   │ 数据管理          │

└──────────┴──────────┴──────────┴──────────┴──────────────────┘
```

### 3.2 数据流全景

**离线链路**：提瓦特API统计 → 生成器（反推个体）→ ODS(JSONL→HDFS) → DWD(MR清洗) → DWS(MR聚合) → ADS(MySQL) → RuoYi API → 分析页面

**实时链路**：练度/满意度生成器(Java loop) → Kafka → Spark Streaming → Redis → RealtimeController API → 数据大屏

### 3.3 数据仓库分层

| 层级 | 存储 | 职责 |
|------|------|------|
| ODS | HDFS/本地 | 原始数据落盘 |
| DWD | HDFS | 清洗+标准化 |
| DWS | HDFS | 轻度聚合 |
| ADS | MySQL | 可视化消费 |
