## 一、项目概述

### 1.1 项目背景

随着《原神》游戏持续运营，每期"深境螺旋"（深渊）的玩家使用率数据成为角色强度评价、版本 Meta 变迁分析的核心依据。国内社区"提瓦特小助手"每期收集数十万份玩家深渊挑战记录，形成了覆盖 60+ 期、100+ 角色的庞大数据集。

本项目基于这些真实统计数据，构建了一套完整的大数据处理与分析平台，涵盖**数据生成 → 采集 → 清洗 → 聚合 → 大屏展示**的全链路闭环。

### 1.2 项目目标

| 目标 | 具体指标 |
|------|---------|
| 数据生成 | 基于真实统计反推生成 200 个虚拟用户的全量角色 BOX 和深渊战绩 |
| 离线计算 | 实现 ODS→DWD→DWS→ADS 四层数据仓库，HDFS 存储 + MapReduce 计算 |
| 实时计算 | Kafka 消费实时抽卡/战斗日志，Spark Streaming 5 秒窗口聚合到 Redis |
| 可视大屏 | RuoYi 框架集成 ECharts，展示使用率红黑榜、角色趋势、配队共现网络 |
| 集群管理 | 自建 6 节点 Hadoop/Spark/Kafka/ZK/MySQL/Redis 全组件集群 |

### 1.3 项目方案版本

项目方案经历了 5 个版本迭代（v1.0 → v5.0），最终确立版本：

- **v5.0 核心架构**：1 条离线链路 + 2 条实时链路，MySQL **仅存 ADS 结果层**
- **关键设计原则**：ODS/DWD/DWS 三层全部在 HDFS，只有 ADS 进入 MySQL 供大屏消费

## 三、数据链路详解

### 3.1 数据生成器（Abyss-Record-Generator）

**核心算法：独立伯努利采样 + 加权随机配队**

```
输入: 提瓦特API统计JSON（115角色 × 100支队伍）
输出: N个用户 × 2个JSON（角色BOX + 深渊战绩）

处理流程（每用户）:
1. 按 own_rate 做伯努利试验决定每个角色是否拥有
2. 从预计算的 3126 对合法配队中筛选该用户可行配队
3. 按 use_rate 乘积加权随机选取 1 对配队（上半+下半）
4. 基于 tier 数据分配合理命座（区分限定五星/常驻/四星）
5. 按 dirty_rate 注入脏数据（10种类型）
```

**关键验证**：verify.js 验证脚本确认聚合偏差 < 0.1%，数据闭环正确。

### 3.2 离线链路四层

#### ODS 层（原始数据层）
- **技术**：Python 脚本
- **输入**：生成器输出的每用户 2 个 JSON
- **处理**：配对合并 → 单文件 JSONL（一行一个用户，含版本标记）
- **输出**：HDFS `/data/abyss/ods/abyss_v66.jsonl`
- **验证点**：200 用户 → 200 行 JSONL

#### DWD 层（明细数据层）
- **技术**：Hadoop MapReduce（Map-only，Jackson JSON 解析）
- **输入**：HDFS JSONL
- **处理**：10 条清洗规则，MultipleOutputs 实现 clean/dirty 分流
- **输出**：
  - `dwd_char_detail`：角色明细 CSV（版本,uid,角色,星级,命座,等级,是否上阵）× 14,635 行
  - `dwd_team_usage`：配队明细 CSV（版本,uid,半场,队伍编号,角色,星级,位置）× 1,424 行
  - `dirty/`：6 类脏数据目录
- **验证点**：200 用户 → 187 通过（93.5%），13 条脏数据

#### DWS 层（汇总层）
- **技术**：Hadoop MapReduce（自定义 Partitioner + Reducer 聚合） + Python 配队共现
- **输入**：DWD char_detail + team_usage
- **处理**：
  - 按 `(version, char_name)` 聚合 own_count/use_count/avg_constellation/avg_level
  - 配队内角色两两共现次数计算
- **输出**：
  - `dws_char_summary`：115 个角色组
  - `dws_team_cooccur`：217 个角色对
- **关键设计**：虚拟键 `__UID__` 巧妙解决 total_users 跨角色共享问题
- **TOP5 使用率**：茜特菈莉 56.68% | 班尼特 56.68% | 希诺宁 53.48% | 玛薇卡 52.41% | 哥伦比娅 44.92%

#### ADS 层（应用数据层）
- **技术**：Python + PyMySQL
- **输入**：HDFS DWS CSV
- **输出**：MySQL `abyss_db` 3 张 ADS 表
  - `ads_meta_ranking`：20 条（红榜 TOP10 + 黑榜 BOTTOM10）
  - `ads_char_trend`：15 条（TOP15 角色版本趋势 JSON 数组）
  - `ads_team_network`：30 条（TOP30 配队共现边）
- **验证点**：3 表共 65 条记录成功写入

### 3.3 实时链路

| 链路 | 数据源 | 中间件 | 计算 | 存储 |

## 六、部署与运维

### 6.1 启动顺序

```
1. Hadoop (start-dfs.sh + start-yarn.sh)
2. ZooKeeper (3节点，至少2节点启动)
3. Kafka (Middleware)
4. MySQL + Redis (Middleware)
5. Spark (start-master.sh + start-slaves.sh)
6. HDFS 数据上传 + MySQL 建表
7. 离线 MR 作业提交
8. Spark Streaming 作业提交
9. Python Producer 启动
10. RuoYi 后端启动 (vserver)
```

### 6.2 一键管道

```bash
# 离线链路全自动
bash scripts/run_offline_pipeline.sh \
  --input /path/to/generated/json \
  --version "v6.6(第一期)"

# 实时链路 Producer
bash scripts/run_realtime_pipeline.sh start

# 全部链路
bash scripts/run_all_pipelines.sh
```

### 6.3 关键问题与解决

| # | 问题 | 解决方案 |
|---|------|---------|
| 1 | Python 3.5 不支持 f-string | 改用 `.format()` |
| 2 | MultipleOutputs 命名限制 | 下划线改连续字符 |
| 3 | YARN 找不到 Jackson | `-libjars` 分发依赖 |
| 4 | master0 无 Maven | javac 直接编译 + curl 下载 jar |
| 5 | Middleware Python 3.5 无 pip | 安装 portable Python 3.10.20 |
| 6 | backup ↔ vserver 主机名互换 | 全网 /etc/hosts 更新 + ZK 分步切换 |

## 七、项目亮点
