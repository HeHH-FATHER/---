## 二、关键技术问题

| 问题 | 根因 | 解决 |
|------|------|------|
| 深渊管道只跑一批就停 | `set -e` + 命令失败 → 脚本退出 | 所有命令加 `|| true`，`set +e` |
| 时效性检查一直 ❌ | `datetime.now()` 无时区 → JS 按本地时区解析差 8h | `utcnow()` + `Z` 后缀 |
| 散点图单角色多点 | API 不区分版本返回全量 | 加 `WHERE version_name` 过滤 |
| 热门角色练度全平权 | 每人保底 1 条 → 60s 窗全相同 | sqrt 压缩权重 + 50 条/秒 |
| 配队名和头像不匹配 | team_index 查 stats 与 members 顺序不一致 | 统一用 members 拼接 |
| 版本顺序反了 | MySQL ORDER BY 默认 ASC → 最新在前 | 前端 `.reverse()` |
| hotChars 12.8s 超时 | 每把武器单查 SQL（40次） | 3 次批量查 |
| API 字段名不对齐 | `latest_five_rate` / `avg_five_rate` | 统一字段名 |
| MW 监控启动不可靠 | SSH 到 MW 不稳定 | 守护进程 `monitor_mw_daemon.sh` |
| Spark 重启读旧 offset | consumer group 复用 | Java 每次用 `System.currentTimeMillis()` |
| 管道 Redis 时间戳未更新 | `set -e` 导致 redis 命令失败即退出 | `|| true` 容错 |

## 三、最终集群状态

```
master0: start_all.sh 一键启动
  ├─ 生成器: Build(50条/s) + Satisfaction(3s/轮)
  ├─ 离线: 抽卡(5min/5000抽/50人/~1.4%) + 深渊(10min/200人/100实数队)
  ├─ Spark: Build + Satisfaction (集群各1核，consumer group每次唯一)
  └─ 监控: monitor_master0.py (6节点)

Middleware:
  ├─ MySQL + Redis + Kafka + ZK
  └─ monitor_collector.py (守护进程保活)

slave1/2/backup: Spark Worker + DataNode
```
